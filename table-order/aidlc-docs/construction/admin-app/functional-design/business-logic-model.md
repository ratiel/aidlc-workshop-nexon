# Admin App — 비즈니스 로직 모델

## 1. 주문 상태 전이 로직

### 상태 정의
```
PENDING (대기중) → PREPARING (준비중) → COMPLETED (완료)
```

### 전이 규칙 (프론트엔드 관점)
| 현재 상태 | 가능한 전이 | 확인 팝업 | API 호출 |
|-----------|------------|-----------|----------|
| PENDING | → PREPARING | 없음 (즉시) | PATCH /api/admin/orders/:id/status |
| PREPARING | → COMPLETED | 확인 팝업 필요 | PATCH /api/admin/orders/:id/status |
| COMPLETED | (최종 상태) | - | - |

### 전이 플로우
1. 관리자가 상태 변경 버튼 클릭
2. (COMPLETED 전이 시) 확인 팝업 표시 → 확인/취소
3. API 호출 (낙관적 업데이트: UI 먼저 변경)
4. 성공: SSE로 다른 클라이언트에 전파
5. 실패: UI 롤백 + 에러 토스트 표시

### 낙관적 업데이트 전략
- 버튼 클릭 즉시 UI 상태 변경 (빠른 반응)
- API 실패 시 이전 상태로 롤백
- 롤백 시 에러 메시지 표시

## 2. 테이블 세션 라이프사이클 관리

### 세션 상태
```
EMPTY (주문 없음) → ACTIVE (주문 있음) → COMPLETED (이용 완료)
                         ↑                        |
                         +------------------------+
                         (새 고객 첫 주문 시 자동 시작)
```

### 이용 완료 처리 플로우
1. 관리자가 "이용 완료" 버튼 클릭
2. 확인 팝업: "테이블 이용을 완료하시겠습니까?"
3. API 호출: POST /api/admin/tables/:id/complete
4. 성공:
   - 해당 테이블 카드의 주문 목록 비우기
   - 총 주문액 0으로 리셋
   - 테이블 상태를 EMPTY로 변경
5. 실패: 에러 토스트 표시, 상태 유지

### 주문 삭제 플로우
1. 관리자가 주문 삭제 버튼 클릭
2. 확인 팝업: "이 주문을 삭제하시겠습니까?"
3. API 호출: DELETE /api/admin/orders/:id
4. 성공:
   - 해당 주문을 목록에서 제거
   - 테이블 총 주문액 재계산
5. 실패: 에러 토스트 표시

## 3. SSE 이벤트 수신 및 상태 동기화

### SSE 연결 관리
```
앱 시작 → SSE 연결 (GET /api/sse/admin)
         → 이벤트 수신 대기
         → 연결 끊김 감지 → 자동 재연결 (지수 백오프)
```

### SSE 이벤트 타입
| 이벤트 타입 | 페이로드 | 처리 로직 |
|------------|---------|-----------|
| `new_order` | Order 객체 | 해당 테이블 카드에 주문 추가, 강조 표시, 총액 업데이트 |
| `order_status_changed` | {orderId, newStatus} | 해당 주문 상태 업데이트 |
| `order_deleted` | {orderId, tableId} | 해당 주문 제거, 총액 재계산 |
| `table_completed` | {tableId} | 테이블 카드 초기화 |

### 재연결 전략
- 연결 끊김 감지 시 즉시 재연결 시도
- 지수 백오프: 1초 → 2초 → 4초 → 8초 → 최대 30초
- 재연결 성공 시: 전체 데이터 새로고침 (GET /api/admin/orders)
- 연결 상태 UI 표시: 🟢 연결됨 / 🟡 재연결 중 / 🔴 연결 끊김

### 신규 주문 강조 로직
- SSE로 `new_order` 수신 시 해당 테이블 카드에 `isHighlighted: true` 설정
- 관리자가 해당 테이블 카드를 클릭하면 `isHighlighted: false`로 해제
- 강조 표시: 카드 테두리 색상 변경 + 펄스 애니메이션

## 4. JWT 인증 플로우

### 로그인 플로우
```
로그인 화면 → 입력 (매장ID, 사용자명, 비밀번호)
            → POST /api/admin/login
            → 성공: JWT 토큰 localStorage 저장 → 대시보드 이동
            → 실패: 에러 메시지 표시
```

### 토큰 관리
- **저장**: localStorage에 JWT 토큰 저장
- **사용**: 모든 API 요청 헤더에 `Authorization: Bearer {token}` 추가
- **만료 감지**: API 응답 401 수신 시 → 토큰 삭제 → 로그인 화면 이동
- **세션 유지**: 브라우저 새로고침 시 localStorage에서 토큰 복원

### 라우트 가드
- 인증 필요 라우트: `/`, `/tables/:id`, `/tables/:id/history`, `/tables/setup`
- 인증 불필요 라우트: `/login`
- 미인증 상태에서 보호 라우트 접근 시 → `/login`으로 리다이렉트
- 인증 상태에서 `/login` 접근 시 → `/`로 리다이렉트

## 5. 대시보드 데이터 로딩 전략

### 초기 로딩
1. 대시보드 진입 시 GET /api/admin/orders (전체 현재 주문)
2. 응답 데이터를 테이블별로 그룹화
3. SSE 연결 시작
4. 이후 SSE 이벤트로 실시간 업데이트

### 테이블 카드 정렬
- 기본 정렬: 최신 주문 시간 순 (가장 최근 주문이 있는 테이블이 먼저)
- 주문이 없는 테이블은 맨 뒤
- 새 주문 수신 시 정렬 자동 업데이트

### 테이블 카드 미리보기
- 각 테이블 카드에 최신 3개 주문 미리보기 표시
- 미리보기 정보: 주문 번호, 메뉴 요약 (첫 메뉴명 + 외 n건), 금액
